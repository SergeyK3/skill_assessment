# Сценарий письменного экзамена (регламенты / ДИ)

Документ задаёт **что нужно сделать на этапе «написание сценария»** и служит основой для реализации (API, бот, веб-чат). Текущий код пакета (`SessionPhase`, Part1/2/3) — про **оценку навыков**; трек **экзамена с протоколом** вводится отдельно и ниже согласован с продуктовыми решениями из [TESTING.md](TESTING.md).

## Цель этапа «writing scenario»

Перевести устное описание процесса в **однозначную спецификацию**: кто что видит, в каком порядке идут шаги, какие тексты и ветвления допустимы, что попадает в протокол. Без этого нельзя стабильно реализовать веб и Telegram одним сценарием.

## Что должно быть на выходе (артефакты)

| # | Артефакт | Назначение |
|---|----------|------------|
| 1 | **Машина состояний** (ниже) | Единая для веб-чата и Telegram; основа для enum/API и тестов. |
| 2 | **Тексты бота** по шагам | Шаблоны с плейсхолдерами `{имя}`, `{срок}`, `{длительность}`, `{название_экзамена}`. |
| 3 | **Блок согласия** | Текст юридического/кадрового согласия при первом обращении + варианты ответа (принять / отказ). |
| 4 | **Параметры экзамена** | Число вопросов (диапазон 3–6), ориентир по времени (15–30 мин), ссылка на папку/раздел регламентов (URL или id в системе). |
| 5 | **Структура вопроса** | Поля: формулировка, ожидаемые темы (для оценки), нужна ли структуризация ответа (да/нет). |
| 6 | **Структура протокола** | Вопросы, ответы (транскрипт/текст), балл по пункту (3–4), **%** (отдельный расчёт, позже — `sentence-transformers`), интегральный балл и %, без проходного порога. |
| 7 | **Завершение для сотрудника** | Благодарность, акцент на сильных сторонах, смысл роли — шаблоны без выставления оценки в диалоге до конца. |
| 8 | **ACL протокола** (на уровне требований) | Кто видит запись: руководитель, кадры, курирующий зам, директор; срок хранения; экспорт PDF. |

После заполнения п. 1–8 можно выносить сценарий в **машиночитаемый формат** (YAML/JSON: шаги, `id`, переходы, `message_key`) — это следующий подэтап разработки, не обязателен для первого черновика в Markdown.

## Машина состояний (черновик)

Зафиксировать минимум такие состояния и переходы (имена — для согласования с кодом позже):

```
[назначен] → приглашение отправлено
     ↓
[ожидание окна доступа] — сотрудник ещё не открыл чат
     ↓
[согласие] — первое открытие канала: показать согласие
     ↓ (принято)              ↓ (отказ → конец / эскалация в HR)
[вводная] — имя, название экзамена, длительность, «готовы начать?»
     ↓ (да)                   ↓ (нет → отложить / напоминание)
[вопрос 1..N] — ответ → транскрипт при необходимости → внутренняя оценка; в диалоге только нейтральная поддержка без баллов
     ↓
[формирование протокола]
     ↓
[выдача протокола] — канал сотруднику + запись объекта в системе с ACL
     ↓
[завершено]
```

Уточнить в сценарии: что происходит при **повторном входе** до конца окна доступа (продолжить с места остановки vs начать заново).

## Шаблоны реплик (заполнить)

- **Приглашение (вне чата):** SMS / email / push — отдельные тексты, единый смысл: срок, цель, где материалы.
- **Старт в чате:** обращение по имени, что начнётся экзамен, 15–30 мин, вопрос о готовности.
- **Между вопросами:** подтверждение важности ответа **без оценки** (как договорено).
- **Финал:** благодарность, позитив, роль в цели организации, прощание; отдельно — доставка протокола.

## Связь с существующим кодом

Сейчас в `skill_assessment` нет отдельной сущности «экзамен по регламентам». На этапе сценария **достаточно** этого документа и таблиц в [TESTING.md](TESTING.md); в коде следующим шагом обычно добавляют:

- тип сессии или отдельную сущность `ExaminationSession` / фазу `EXAM_REGULATION`;
- таблицы протокола и согласия;
- адаптеры каналов (веб / Telegram) с общим движком состояний.

## Чек-лист «готово ли написание сценария»

- [ ] Все состояния и исключения (отказ, таймаут окна, повторный вход) описаны.
- [ ] Тексты согласия согласованы с юридическим/кадровым контуром (даже черновик).
- [ ] Известен состав ролей для ACL протокола и нужен ли экспорт PDF в MVP.
- [ ] Для каждого из 3–6 вопросов есть черновик формулировки или правило выбора из банка.
- [ ] Зафиксировано: оценка **не** озвучивается в диалоге до финала; протокол — полный источник правды.

## Пробелы и вводные вопросы

Актуальные **ответы заказчика**, пояснения терминов (шаблоны, банк вопросов, инкремент PDF) и оставшиеся пункты: [EXAM_OPEN_QUESTIONS.md](EXAM_OPEN_QUESTIONS.md).

Кратко: назначение экзамена — **руководитель ОК** по указанию **директора/зам по адм. вопросам** (при отсутствии руководителя ОК — **директор или зам**); фазы могут идти **по разным каналам**, веб+Telegram **параллельно**; в веб — **персональная ссылка**; прерывание — **продолжить**; отказ от согласия — **блок до HR**, при следующем входе — **снова согласие**; **один сценарий**, **фиксированный список вопросов** в MVP, **шаблоны позже**; в протоколе — **транскрипт**, не аудио; **PDF** — не обязателен в первой поставке с сессией, **допустимо в течение суток**; хранение **1 год**, выгрузка и очистка сервера; в MVP протокол минимум у **руководителя ОК**; идентификаторы — **`employee_id` / `client_id` из ядра**.
