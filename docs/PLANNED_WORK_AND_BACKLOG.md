# План работ (backlog): согласия, история сессий, результаты, напоминание

Документ фиксирует договорённости и **to-do**, чтобы не потерять направления развития после текущих правок (Telegram, календарь, должность из HR).

---

## 1. Согласие на обработку ПДн (второе сообщение / этап)

**Идея:** после того как сотрудник согласился на участие и выбрал дату/время экзамена (опроса), **дополнительно** зафиксировать явное согласие на обработку персональных данных — отдельным сообщением в боте (или сразу после выбора слота, или по расписанию до старта — **уточнить порядок** с юристом/HR).

**Сделано (MVP):** после выбора времени в календаре отправляется второе сообщение с запросом согласия; ответы «да/нет» обрабатываются в `telegram_docs_survey_consent.py`, статус и время пишутся в `sa_assessment_sessions` (`docs_survey_pd_consent_*`, слот в `docs_survey_scheduled_at`). Нужен `TELEGRAM_ENABLE_POLLING=1`.

- [ ] При необходимости согласовать формулировки с юристом; при желании — общий шаблон с экзаменом по регламентам (`telegram_examination`).
- [ ] UI/API для просмотра статуса согласия по сессии (по мере появления истории сессий).

---

## 2. История назначений сессий (UI + данные)

**Идея:** после **выбора сотрудника** в рабочей сессии показывать **историю назначений** (и при необходимости — блок «новая сессия») **перед** основным интерфейсом этапов.

**Функции:**

- [ ] Хранить и отображать историю назначений сессий по сотруднику/организации.
- [ ] Поле/колонка **когда запланирована новая сессия** (дата/время из календаря бота, если сохраняем).
- [ ] Возможность **отказаться от дальнейшего назначения** новых сессий (например, HR видит, что «достаточно» попыток).
- [ ] Таблица **«История назначений сессий»** с:
  - **фильтрами**;
  - **должностью**;
  - **подразделением**;
  - **датой** сессии / назначения.
- [ ] В **модальном окне**: просмотр результатов по строке, **редактирование**, **удаление** (с правами и подтверждением).

**Зависимости:** сохранение слота из Telegram в БД (если ещё не везде записано), модель «назначение vs сессия».

---

## 3. Таблица результатов оценки (опрос / кейс / руководитель)

**Идея:** отдельная (или расширенная) таблица **итогов оценки** с похожей структурой:

- [ ] Фильтры, **должность**, **подразделение**, **даты**.
- [ ] Итоги по этапам: **опрос по документам**, **кейс**, **оценка руководителя** (и при необходимости сводный результат).
- [ ] Согласованность с колонками «Док. / Кейс / Рук.» в текущем UI и с API отчётов.

---

## 4. Сценарий «наступило время + напоминание за 30 минут + готовность отвечать»

**Идея:** смоделировать и задокументировать полный поток:

- [ ] За **30 минут** до запланированного слота — **напоминание** в Telegram.
- [ ] Сотрудник открывает бот, подтверждает **готовность** к прохождению.
- [ ] Далее — вопросы, голос/текст, фиксация ответов (как обсуждалось ранее).

**Задачи:**

- [ ] Планировщик задач (APScheduler / cron / очередь) + хранение `scheduled_at` по сессии.
- [ ] Сообщения бота и состояния сессии (state machine).
- [ ] Короткий **документ сценария** (можно расширить `EXAM_SCENARIO.md` или отдельный `docs/PART1_TELEGRAM_SCENARIO.md`).

---

## Порядок внедрения (рекомендация)

1. Данные: слот из бота → БД, история сессий.  
2. UI: история + отказ от новых назначений + модалка.  
3. Согласие ПДн — второй шаг в боте после слота.  
4. Напоминание за 30 минут и сценарий «готов».

Приоритеты можно менять после оценки трудозатрат и юридических требований к согласию.
